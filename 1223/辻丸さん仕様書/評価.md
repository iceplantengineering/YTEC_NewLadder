## 3. 不完全な反映により残る問題

### 🔴 重大な問題

#### 3.1 信号機交渉の排他制御不全
**問題**: A01のX001～X006信号機要求時に、他局からの同時要求を排除できない

**影響**:
- 複数搬送時に同時に信号機要求を出し、合意形成に失敗
- AGVが永遠に待機状態のまま放置される

#### 3.2 使用中フラグの単局設計
**問題**: B01/B02のM3070/M4070が各局ローカルのみ

**影響**:
- 局間で同じコンテナを同時に処理しようとする
- WS間で競合が発生し、どちらも進行不能に陥る

### 🟡 中程度の問題

#### 3.3 AGV連携のタイムラップ
**問題**: B01仕様書でAGV要求→搬送指示の順序保証が不十分

**影響**:
- AGV搬送要求を出す前にクレーン側が処理完了
- 空き容量通知がAGVに届かず、過剰在庫が発生

#### 3.4 状態整合性チェックの欠如
**問題**: A02の入庫フラグM2040とB01の使用中フラグM3070の整合性未確認

**影響**:
- クレーン側は「入庫完了」と判断
- コンベア側は「使用中」のまま処理継続
- データ不整合により緊急停止が発生

## 5. 復旧時の安全性分析

### 5.1 危険性評価結果: 🔴 **事故の危険性あり**

### 5.2 危険シナリオと発生条件

#### 【危険シナリオ1】タイムアウト強制解除後の突然再起動

**A02仕様書（スタッカークレーン本体）の問題点**:

| ラダー番号 | 問題箇所 | 危険性 |
|------------|----------|--------|
| **R0045～R0050** | T001タイムアウト後、即座にM2030（移動中）リセット | サーボが慣性で動いている中でフラグ消去 |
| **R0120～R0130** | T005タイムアウト後、M2050（叉出中）強制解除 | フォークが伸出中のまま復帰命令 |
| **R0080～R0095** | 復帰シーケンスにオペレータ確認なし | 自動で原点復帰開始 |

**具体的な事故シナリオ**:
```
［状況］叉出中にT005タイムアウト発生
    ↓
［A02仕様書の動作］M2050強制解除 → M2060（復帰中）セット
    ↓
［危険］フォークが倉内に伸出した状態で、復帰動作開始
    ↓
［結果］フォークが棚やコンテナと衝突
```

#### 【危険シナリオ2】デッドロック解除後の未完了処理再実行

**A01仕様書の問題点**:

| ラダー番号 | 問題箇所 | 危険性 |
|------------|----------|--------|
| **R0180～R0195** | T000タイムアウト後、M2010（処理要求）再セット | 前回の処理完了を確認せず再実行 |
| **R0220～R0235** | 強制解除後、即座に次の要求許可 | 2重搬送命令発行の可能性 |

**具体的な事故シナリオ**:
```
［状況］入庫処理中にデッドロック発生 → T000タイムアウト
    ↓
［A01仕様書の動作］M2010再セット → M2000（入庫中）維持
    ↓
［危険］A02クレーンは「入庫中断」、A01は「入庫中」のまま
    ↓
［結果］クレーンが勝手に再起動し、コンテナを二重に搬入しようとする
```

#### 【危険シナリオ3】B01/B02コンベアの再起動リスク

| 仕様書 | 問題箇所 | 危険性 |
|--------|----------|--------|
| **B01** | T001タイムアウト後、M3040（搬送中）強制解除 | コンベア上のコンテナ放置 |
| **B02** | T002タイムアウト後、M4040（搬送中）強制解除 | 搬送方向が逆転する可能性 |

### 5.3 安全対策の不足している箇所

| 安全対策 | 現状 | 必要性 |
|----------|------|--------|
| **復旧前オペレータ確認** | なし | **必須** |
| **原点位置確認** | なし | **必須** |
| **サーボ駆動禁止状態の解除** | なし | **必須** |
| **未完了処理のクリア確認** | なし | **必須** |
| **手動復旧モードへの切り替え** | なし | **推奨** |

### 5.4 推奨される安全対策ロジック

```pascal
// タイムアウト発生時に実装すべき安全ロジック（例）

NETWORK: タイムアウト処理
LD      T001                    // タイムアウト発生
OUT     M9000                   // 復旧要求フラグ

NETWORK: 復旧確認待ち
LD      M9000
ANDN    X900                    // オペレータ確認スイッチ
OUT     M9001                   // 復旧待機中

NETWORK: 復旧実行
LD      M9001
AND     X900                    // オペレータ確認あり
AND     M8000                   // サーボ準備完了
ANDN    M2030                   // 移動中でない
ANDN    M2040                   // 入庫中でない
ANDN    M2050                   // 叉出中でない
OUT     M9002                   // 復旧実行許可

NETWORK: 状態クリア
LD      M9002
RST     M2000                   // 入庫中フラグクリア
RST     M2010                   // 処理要求クリア
RST     M2030                   // 移動中フラグクリア
SET     M2060                   // 復帰モードへ
```

## 7. 結論と提言

### 7.1 結論

1. **デッドロック対策**: 4件の仕様書はデッドロック対策を部分的に反映していますが、**排他制御フラグと局間通信に関する重要な対策が不完全**です。これらを追加実装しない場合、複数搬送時やAGV連携時にデッドロックが再発する可能性があります。

2. **安全性**: 現行仕様書のままでは、デッドロック復旧時にスタッカーが走り出す事故の危険性が**高い**と言えます。オペレータ確認と原点確認なしに自動復帰する設計になっているため、**緊急安全対策の追加が強く推奨されます**。

### 7.2 優先的に対応すべき事項

| 優先度 | 対策内容 | 対象 | 理由 |
|--------|----------|------|------|
| **最優先** | 復旧前オペレータ確認実装 | 全体 | 事故防止 |
| **最優先** | M2046排他フラグの実装 | A02 | デッドロック防止 |
| **最優先** | 使用中フラグの全局共有化 | B01/B02 | 競合防止 |
| **高** | 信号機交渉時の排他制御追加 | A01 | AGV待機防止 |
| **高** | 原点位置確認の実装 | A02 | 衝突防止 |
| **中** | AGV要求信号の順序保証 | B01 | 過剰在庫防止 |
| **中** | WS間状態整合性チェック実装 | 全体 | データ不整合防止 |
